Treat this page as the reading map for Release Runbook, not as a raw repo mirror.

Read this when you need to understand shipping discipline, not just feature behavior—especially if you own versioning, packaging, or release communication.

What this document actually covers

Release runbook docs explain how changes move from implementation to shippable output with checks, sequencing, and rollback awareness.

Questions this page should answer fast

  • What has to be true before a release is considered ready?
  • Which checks are release-critical and which are merely useful context?
  • How should release docs stay aligned with install docs, changelog expectations, and operational follow-up?

What to focus on inside the upstream doc

  • Release gates, sequencing, and evidence of readiness instead of vague confidence language.
  • The transition from local success to distributable artifact and public update path.
  • Rollback and post-release verification as first-class parts of the flow.
  • Where release docs rely on architecture, config, and install assumptions.

Common mistakes

  • Treating release as packaging only, with no operator validation path.
  • Skipping rollback documentation because the release seems small.
  • Letting release notes and install expectations drift apart.

Recommended reading order

Read by problem type first, then branch back into install, config, modes, MCP, or troubleshooting only when the next step is clear.

Step 1

Start from the operator trigger

Frame the runbook around the change, failure, or release event that brings you here.

Step 2

Look for checks and reversibility

Highlight the points where validation, rollback, and evidence matter most.

Step 3

Tie the procedure back to live state

Make sure the documented steps still match current install, config, and runtime behavior.
