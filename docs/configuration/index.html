DeepSeek TUI Configuration Docs Guide

Best Use

Treat this page as the reading map for Configuration Docs, not as a raw repo mirror.

Read this when the app starts but behaves differently from what you expect, or when you need to make settings portable across machines and providers.

What this document actually covers

Configuration docs explain which settings change provider behavior, runtime defaults, memory, tools, and environment-specific assumptions.

Questions this page should answer fast

  • Which settings are required, optional, experimental, or provider-specific?
  • Which values belong in config files versus environment variables?
  • How do you change defaults without making the setup brittle for teammates or future upgrades?

What to focus on inside the upstream doc

  • Provider blocks, model-related defaults, and how per-provider overrides interact with global settings.
  • Settings that affect tool permissions, memory behavior, or orchestration rather than only cosmetics.
  • The order of precedence between checked-in config, local overrides, and shell-level environment.
  • A minimal safe baseline before you start tuning advanced options.

Common mistakes

  • Mixing provider-specific keys into a global mental model.
  • Changing many defaults at once and losing a known-good baseline.
  • Treating environment variables as documentation instead of reproducible config.

Read by problem type first, then branch back into install, config, modes, MCP, or troubleshooting only when the next step is clear.

Where this page sits in the site

These docs detail pages are not just repo mirrors. Their job is to tell you why a topic matters, which parts deserve attention first, and which branch of the site should take over once the document boundary is clear. For a project that spans install, configuration, modes, tools, and operator workflows, that route-first structure is more useful than a flat Markdown mirror.

When to leave this page

Once you know whether the next action belongs in install, config, modes, MCP, troubleshooting, or hands-on guides, move there quickly. The detail page is most useful when it gives you a sharper reading order and a better mental boundary for the topic, not when it tries to absorb every downstream action page.
