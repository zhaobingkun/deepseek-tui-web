排错总览

CodeWhale 的常见问题，大多还是安装、路径和配置基础问题

很多从 DeepSeek TUI 旧名字搜进来的人，最后遇到的仍然是命令没进 path、provider 没选对、API Key 没生效这些基础问题。

优先级建议

先查命令能不能调用，再查配置和 provider，最后再怀疑更复杂的问题。

大多数排错都不需要从最复杂的方向开始。

按问题类型选择排错路径

先把这页当成排错列表。先判断是命令路径、provider、二进制还是 MCP 问题，再开详情页。

命令

我装好了，但命令还是调不起来

如果问题是 command not found 或 shell path，先走命令修复路线，不要先查高级配置。

去命令修复
Provider

命令能跑，但 provider 还是不对

如果命令存在，但鉴权、endpoint 或返回行为不对，先走 provider 排错路线。

去 provider 排错
进阶

我怀疑是 MCP 或二进制安装这层的问题

如果基础命令和配置看起来没问题，再看 release binaries 或 MCP 这一层。

去 MCP 排错

排错文章列表

先把问题缩小到具体一层，再看对应修复页，不要把安装、provider 和 MCP 调试混在一起。

Command Not Found

command not found 修复

安装后最常见的第一类错误，通常和 shell path 或安装路径有关。

打开修复页
Homebrew

Homebrew command not found

如果 brew 说装好了，但终端还是找不到命令，这页更对应。

打开 Homebrew 修复
Provider

Provider 排错

打开 provider 排错
Binaries

Release binaries

打开 release binaries
API Key

API Key 与 provider

如果命令存在但无法使用，配置线通常比安装线更关键。

打开 API Key 页
MCP

MCP 排错

如果你怀疑是 MCP 坏了，这页会先帮你分清楚是不是基础配置没稳。

打开 MCP 排错
